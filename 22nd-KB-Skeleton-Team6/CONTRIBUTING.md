# Git 규칙

# 🤝 협업 가이드 (Contributing Guide)

이 프로젝트에 참여해주셔서 감사합니다! 우리 팀의 일관된 코드 관리와 효율적인 협업을 위해 아래 규칙을 준수해 주세요.

---

## 1. 작업 프로세스 (Workflow)

모든 작업은 아래 순서를 따릅니다.

1. **Issue 생성**: 작업 시작 전, Issue를 생성하여 작업 내용을 정의합니다.
2. **Branch 생성**: 생성된 Issue 번호를 기반으로 새로운 브랜치를 만듭니다.
3. **작업 및 Commit**: 규칙에 맞게 커밋 메시지를 작성합니다.
4. **Pull Request (PR) 생성**: 작업 완료 후 `main` 브랜치로 PR을 보냅니다.
5. **Code Review**: 팀원의 리뷰를 받고 승인(Approve)을 얻습니다.
6. **Merge**: `Squash and Merge` 방식으로 메인 브랜치에 합칩니다.

---

## 2. 브랜치 전략 (Branching Strategy)

우리는 **이름과 작업 유형**을 포함한 브랜치 명명 규칙을 사용하여, 누가 어떤 작업을 하고 있는지 명확히 구분합니다.

- **main**: 배포 가능한 최신 상태의 코드입니다. (직접 Push 절대 금지 ❌)
- **작명 형식**: `이름/타입-#[이슈번호]-기능명`
  - **이름**: 본인의 이름(또는 닉네임, 이니셜)을 소문자로 작성합니다.
  - **타입**: 작업의 종류 (feat, fix, docs 등)
  - **예시**:
    - `sunghoon/feat-#12-login-api`
    - `jiyun/fix-#45-header-typo`
    - `kildong/docs-#2-readme-update`

**Tip:** 브랜치 이름에 본인 이름이 들어가면, 작업 중 혼선이 생겼을 때 바로 누구에게 물어봐야 할지 알 수 있어 소통이 빨라집니다!

---

## 3. 커밋 메시지 규칙 (Commit Convention)

커밋 메시지는 작업 내용을 파악하기 쉽게 아래 형식을 지킵니다.

**형식:** `타입: [이슈번호] 요약 설명`

| 타입       | 의미                                           |
| :--------- | :--------------------------------------------- |
| `feat`     | 새로운 기능 추가                               |
| `fix`      | 버그 수정                                      |
| `docs`     | 문서 수정 (README, CONTRIBUTING 등)            |
| `style`    | 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음) |
| `refactor` | 코드 리팩토링                                  |
| `test`     | 테스트 코드 추가 및 수정                       |
| `chore`    | 빌드 업무, 패키지 설정 수정 등                 |

**예시:** `feat: [#12] 구글 소셜 로그인 연동`

---

## 4. 이슈(Issue) 및 PR 연결

프로젝트의 히스토리 관리를 위해 모든 PR은 이슈와 연결되어야 합니다.

- **이슈 연결**: PR 본문에 `Closes #이슈번호` 또는 `Fixes #이슈번호`를 기재합니다.
- **자동 종료**: PR이 `main`에 머지되면 연결된 이슈는 자동으로 닫힙니다.
- **작업 범위**: 하나의 PR은 가급적 하나의 이슈만 해결하도록 작게 유지합니다.

---

## 5. 머지 전략 (Merge Strategy)

- **Squash and Merge**: 우리 팀은 이 방식을 기본으로 합니다.
  - 여러 개의 커밋 로그를 하나의 깔끔한 커밋으로 합쳐서 `main`에 기록합니다.
  - `main` 브랜치의 히스토리를 기능 단위로 깨끗하게 유지하기 위함입니다.

---

## 6. 코드 리뷰 규칙 (Code Review)

- **리뷰어 지정**: PR 생성 시 팀원 전체 혹은 담당자를 리뷰어로 지정합니다.
- **승인 조건**: 최소 **1명 이상의 Approve**가 있어야 머지가 가능합니다.
- **피드백**: 코드의 논리적 오류, 가독성, 성능 개선안 등을 자유롭게 소통합니다. 질문이나 제안은 언제나 환영입니다!

---

## 7. 기타 주의사항

- **.gitignore**: 보안 파일(`.env`), 개인 설정 파일, 빌드 결과물(`dist/`, `node_modules/`)은 절대 올리지 않습니다.
- **코드 스타일**: [예: Prettier / ESLint] 설정에 따라 코드 포맷을 맞춘 후 커밋합니다.
